# Micro-framework mandatory rules

Every flow that adopts the micro-framework must comply with these 10 rules. Each has a
binary verification criterion: **pass / fail**.

Use `checklist-arquitectura.md` to verify before versioning any to-be flow.

---

| ID | Rule | Verification criterion |
|----|-------|--------------------------|
| REG-001 | No flow has hardcoded credentials in nodes | The flow's exported JSON contains no tokens, API keys, or passwords as literal values |
| REG-002 | Every flow has a `run_id` from E1 that propagates to all subflows | The `run_id` field appears in every subflow's output and in every log entry |
| REG-003 | Every orchestrator flow has `errorWorkflow` configured in its settings | The JSON's `settings.errorWorkflow` field is not empty |
| REG-004 | External integrations have retry enabled | The HTTP Request node has `options.retry.enabled: true` |
| REG-005 | Database writes include idempotency control | The insert query includes `ON CONFLICT ... DO NOTHING` or equivalent |
| REG-006 | Every stage emits a structured JSON log | The Code node of every stage includes `console.log(JSON.stringify({...}))` with the minimum fields |
| REG-007 | Business logic is isolated in E2 | The E2 subflow contains no HTTP Request nodes or database nodes |
| REG-008 | External integrations live in E3 | HTTP Request and database nodes are exclusively in E3 and E4 |
| REG-009 | The orchestrator flow responds with appropriate HTTP status codes | 200 for success, 400/422 for invalid input, 401 for failed authentication |
| REG-010 | Every flow has at least one documented ADR | The case's `adr/` folder contains at least one versioned ADR file |

---

## Rule detail

### REG-001 — No hardcoded credentials

**Why:** Credentials in the exported JSON are exposed in the Git repository, compromising
the security of external systems.

**How to verify:** Search the exported JSON for the strings: `password`, `token`,
`api_key`, `Bearer`, `secret`. None should have a literal value; they must reference n8n
credentials.

**In n8n:** Use n8n's Credentials manager. The exported JSON will only contain the
credential's name, not its value.

---

### REG-002 — run_id propagated

**Why:** `run_id` allows correlating all logs of an execution without depending on n8n's
internal history.

**How to verify:** Review the JSON output of every subflow. The `run_id` field must be
present with the same value in all of them.

**Format:** `RUN-BOT-{timestamp}-{random6}` or `RUN-IOT-{timestamp}-{random6}`

**Note on as-is:** This format applies to the to-be. The as-is run-logs use a simplified
format `{case}-{version}-{set}-{index}-{hash}` generated by the measurement harness
(`automatizacion/run_corridas.py`) because the as-is flows violate REG-002 by design (they
don't propagate `run_id` internally). See `docs/protocolo-evidencias.md` §5.

---

### REG-003 — errorWorkflow configured

**Why:** Without an error flow, failures are operationally invisible and are never
recorded.

**How to verify:** In the orchestrator's JSON, the `settings.errorWorkflow` field must not
be an empty string.

---

### REG-004 — Retry enabled on HTTP

**Why:** External services fail transiently. Without retry, a single timeout results in
lost data or tickets.

**Minimum configuration on the HTTP Request node:**
```json
"options": {
  "retry": {
    "enabled": true,
    "maxRetries": 3,
    "waitBetweenTries": 2000
  }
}
```

---

### REG-005 — Idempotency on writes

**Why:** Retries silently create duplicate records if there is no idempotency control.

**Required SQL:**
```sql
INSERT INTO tabla (idempotency_key, ...)
VALUES ($1, ...)
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING id;
```

---

### REG-006 — Structured JSON log per stage

**Why:** n8n's execution history is not programmatically queryable. JSON logs allow
computing latencies and correlating events.

**Minimum fields per stage:**
- E1: `run_id`, `etapa`, `status`, `errores`, `start_ts`
- E2: `run_id`, `etapa`, `status`, `resultado_clave`, `duracion_ms`
- E3: `run_id`, `etapa`, `status`, `idempotency_key`, `duracion_ms`
- E4: `run_id`, `etapa`, `status`, `notificacion_enviada`

---

### REG-007 — Business logic only in E2

**Why:** Separating domain from integrations allows modifying business rules without
touching adapters, and testing E2 in isolation.

**Forbidden in E2:** HTTP Request nodes, Postgres nodes, database nodes of any kind.

---

### REG-008 — Integrations only in E3/E4

**Why:** Centralizing integrations makes it easier to replace external services without
modifying business logic.

**E3:** Persistence adapters and external APIs.
**E4:** Notification channels and the HTTP response to the client.

---

### REG-009 — Appropriate HTTP status codes

**Why:** Incorrect status codes make diagnosis harder and break the contract with the
webhook client.

| Situation | Status code |
|-----------|-------------|
| Success | 200 OK |
| Invalid input (missing fields, wrong types) | 400 Bad Request |
| Semantically invalid input (ranges, contracts) | 422 Unprocessable Entity |
| Invalid authentication token | 401 Unauthorized |
| Uncontrolled internal error | 500 Internal Server Error |

---

### REG-010 — At least one ADR per case

**Why:** Undocumented architectural decisions get lost and make ATAM evaluation harder.

**Where:** `casos-de-estudio/{case}/adr/ADR-001-{name}.md`

**Template:** See `microframework/plantillas/ADR-plantilla.md`

## REG-* → ISO/IEC 25010 mapping

Each rule contributes to one or more ISO/IEC 25010 quality-model attributes. This mapping
feeds the ATAM evaluation (PHASE 7) and each case's traceability matrix.

| Rule | ISO 25010 characteristic | Sub-characteristic | Justification |
|---|---|---|---|
| REG-001 | Security | Confidentiality | Prevents secret exposure in the versioned JSON |
| REG-002 | Maintainability | Analyzability | `run_id` allows correlating logs without depending on n8n's internal history |
| REG-003 | Reliability | Fault tolerance | `errorWorkflow` captures failures and makes them operationally visible |
| REG-004 | Reliability | Recoverability | Retry absorbs transient failures of external services |
| REG-005 | Reliability | Maturity | Idempotency prevents duplicates on retries |
| REG-006 | Operability | Monitorability | Structured JSON log enables computing MTTD and per-segment latencies |
| REG-007 | Maintainability | Modularity | Isolating the domain reduces change impact |
| REG-008 | Maintainability | Modularity / Reusability | Centralizing integrations allows replacing services without touching the domain |
| REG-009 | Functional suitability | Correctness | Correct status codes fulfill the HTTP contract with the client |
| REG-010 | Maintainability | Analyzability | ADRs document decisions for auditing and ATAM |

Cross-reference: see `docs/context/proyecto-overview.md`, section "ISO/IEC 25010 mapping",
for the link to the thesis proposal's metrics.
