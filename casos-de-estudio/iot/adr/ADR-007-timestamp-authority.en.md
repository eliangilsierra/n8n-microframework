# ADR-007 — Timestamp authority: use the sensor's (client) timestamp

**Date:** 2026-05-01
**Status:** Implemented 2026-05-02
**Quality attribute:** Reliability / Maturity (ISO/IEC 25010)
**Related rules:** REG-005, IoT ADR-003 (sensor-timestamp idempotency)
**ATAM scenario:** IOT-Q3 (resending the same reading → 0 duplicates)

---

## Context

Every sensor reading includes a `timestamp` field generated by the sensor itself. The n8n
system also has its own clock (`new Date().toISOString()` in the Code node).

The IoT case's idempotency key is `{sensor_id}_{timestamp}` (documented in IoT ADR-003).
This decision means the timestamp used in the key determines whether two submissions of
the same data are treated as the same reading (idempotent) or as different readings
(duplicate).

The question is: use the **sensor's** timestamp (from the input payload) or the
**server's** timestamp (n8n, at processing time)?

---

## Decision

The **sensor's timestamp** (from the payload's `timestamp` field) is used as the temporal
authority for the idempotency key and for the database record.

### Technical justification

If the server's (n8n) timestamp were used:

```
Sensor's 1st send:  timestamp_payload=2026-05-01T10:00:00Z → server=10:00:01Z → key="sensor1_10:00:01Z"
Sensor's retry:      timestamp_payload=2026-05-01T10:00:00Z → server=10:00:03Z → key="sensor1_10:00:03Z"
```

→ The two sends of the same data generate different keys → they're inserted as two
different records → silent duplicate. ADR-003's idempotency would be rendered inert.

With the sensor's timestamp:

```
Send 1:   timestamp_payload=2026-05-01T10:00:00Z → key="sensor1_2026-05-01T10:00:00Z"
Retry:    timestamp_payload=2026-05-01T10:00:00Z → key="sensor1_2026-05-01T10:00:00Z"
```

→ Both keys are identical → `ON CONFLICT DO NOTHING` → 0 duplicates. ✓

### Validation against clock drift

E1 validates that the sensor's timestamp is not more than 5 minutes in the future:

```javascript
const ahora = new Date();
const ts_sensor = new Date(payload.timestamp);
const diff_ms = ts_sensor - ahora;

if (diff_ms > 5 * 60 * 1000) {
  errores.push(`timestamp en el futuro: diferencia de ${Math.round(diff_ms/1000)}s (máximo 300s)`);
}
```

This validation detects sensors with a desynchronized clock (drift) that could generate
incorrect idempotency keys for non-real future readings.

---

## Alternatives considered

| Alternative | Reason for rejection |
|-------------|-----------------|
| Server (n8n) timestamp | Breaks idempotency: two sends of the same payload will have different keys and be inserted as duplicates |
| UUID generated by E1 (independent of the timestamp) | Not idempotent: every E1 invocation generates a different UUID, even for the same payload |
| Hash of the full payload as the key | Robust but computationally more expensive and sensitive to differences in non-significant fields (e.g., spaces in `location`) — the `{sensor_id, timestamp}` key is sufficient and more readable |
| Server timestamp + idempotency field in the HTTP header | Requires the client (sensor) to send an idempotency header — modifies the input contract |

---

## Consequences

**Positive:**
- Idempotency guaranteed: multiple sends of the same payload produce exactly 1 record in
  PostgreSQL, verifiable with `SELECT COUNT(*) FROM lecturas_sensor WHERE idempotency_key = '...'`.
- The `timestamp` in the DB reflects when the measurement occurred, not when it reached
  the system — semantically more correct for time-series analysis.
- Input Set K (resending the same reading) → 0 duplicates, directly measurable.

**Negative:**
- Sensors with a desynchronized clock are rejected with 422 (E1 validates the drift). This
  can be a problem if the physical sensors don't have NTP configured. Declared as a lab
  environment limitation (the datasets simulate recent, consistent timestamps).
- Historical readings (timestamp far in the past) are not rejected by E1 — they could be
  inserted with an old timestamp. Declared as outside the case study's scope.

---

## Verification criterion

1. Input Set K (same payload sent twice) → `SELECT COUNT(*)` in PostgreSQL = 1 (REG-005)
2. Payload with `timestamp` 10 minutes in the future → HTTP 422, `errores` contains
   "timestamp en el futuro"
3. `idempotency_key` in the table = `{sensor_id}_{sensor's timestamp}` (not the server's
   timestamp)
